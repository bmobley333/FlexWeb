Google Apps Script Caching & Storage Comparison
Feature
	Session Cache (Global Variables)
ONLY readable by the single Sheet/Doc and all of its GAS including FlexLIb but OTHER sheets/Docs cannot read that documents Session Cache (e.g. there isn’t a "Session Cache” that spans FlexLib!. 
Can hold most javascript data types.
	PropertiesService
ONLY readable by the script or the document (two separate cases - see below)
Key Value pairs and of strings only
	CacheService
getScriptCache(): Version is all GAS files within ONE Doc/Sheet’s set of GAS - so limited to JUST CS or Just Codex, etc.
getUserCache() version readable by ALL Google Scripts from ANY source even 3rd party - fragile.
	Analogy
	🧠 Short-term memory / Whiteboard
	🔒 Sticky notes / Lockbox
	🗄️ Shared, temporary locker
	Longevity
	Volatile: Lasts only for the current session. Wiped on close, reload, or ~30 mins of inactivity.
	Permanent: Data persists until explicitly deleted by a script.
	Semi-persistent: Guaranteed for 20 mins, max 6 hours. Can be cleared by Google anytime after 20 mins.
	Storage Size
	Large: No hard limit, tied to script execution memory (having nothing to do with GAS code execution ending). Good for thousands of rows.
	Small: 500 KB total storage. 9 KB per single value.
getScriptProperties() (per script): Each script project gets its own private 500 KB of storage.
* Your FlexLib script has its own 500 KB.
* Your CS script has its own separate 500 KB.
* Your DB script has its own separate 500 KB.
* This is why the scripts couldn't see each other's properties—they each have their own private storage space.
getDocumentProperties() (per document): Each individual spreadsheet or document file gets its own 500 KB of storage that is shared by any script running on that document.
* Character Sheet A has its own 500 KB.
* Character Sheet B has its own separate 500 KB.
* The Codex has its own separate 500 KB.


	Medium: 10 MB total storage. 100 KB per single value.
	Scope / Accessibility
	Per User, Per Document, Per Session: Completely private and isolated to a single user's active session in a specific file.
	Has three distinct scopes: • getScriptProperties(): Private to the script project. • getDocumentProperties(): Shared by all scripts running on that specific document. • getUserProperties(): Shared by all scripts authorized by that specific user across all their documents.
	Has two scopes: • getScriptCache(): Private to the script project. • getUserCache(): Private to the user, accessible by any script they run.
	Overwrite Risk
	None: Completely isolated to the single session.
	Varies by Scope: • Script: None. • Document: Low. Possible if another app on the same sheet uses the same keys. • User: Highest risk. Any authorized app could potentially use the same key.
	Varies by Scope: • Script: None. • User: High risk, similar to User Properties.
	Best For
	Storing data needed repeatedly within a single session for maximum speed (like our onEdit optimization).
	Storing small, simple, permanent settings like user preferences or configuration flags.
	Caching the results of slow operations (like web requests) that might be needed again within a few hours.

Project Goals for TTRPG: Flex
My goal is to create a Google Apps Script-powered system for my TTRPG, Flex, using a suite of interconnected Google Sheets and Docs. The core components are:
* Player’s Codex: The central dashboard for players and GMs. It manages file setup, character creation, and version control. It will contain its own internal data sheet for configuration and a Log sheet for troubleshooting.
* Character Sheet: A dedicated Google Sheet for a single character. For optimal performance, it will contain hidden sheets with cached data copied directly from the relevant databases, eliminating the need for live =IMPORTRANGE() calls during gameplay.
* Core DB: A Google Sheet database containing all the standard rules, abilities, items, and tabular information for the game.
* Campaign DB: A Google Sheet database template for a Game Master (GM) to copy and customize with their own unique campaign content.
* Rules: A master Google Doc containing the game's rules.
* Version Tracker: The definitive source of truth for the entire system. This simple Google Sheet will track game versions and store the master File IDs for all other components.
The Player’s Codex will be the main console. A player's first action will be to run an initial setup script that copies the master Core DB and Character Sheet template into their personal Google Drive. GMs will also copy the Campaign DB. This architecture is designed to distribute the workload away from a central account and onto each player's individual Google Account, ensuring the system remains functional and performant on the free tier.
When a player creates a new character, the Player's Codex will duplicate their local Character Sheet template and then automatically cache all data from their Core DB (and a GM's Campaign DB, if linked) into hidden sheets within that new character file. GMs will be able to easily retrieve their Campaign DB's file ID from their Codex to share with players, who can then link their characters to that custom dataset.
The Player’s Codex will track the version each character is built on. Using a time-driven trigger, it will check the master Version Tracker monthly for updates. If a new version is available, it will download updated copies of the core files for creating new characters. New characters can use the latest data at any time using a dedicated "Refresh Data" function.
To maintain simplicity and accessibility, all Google Apps Script code will be bound directly to the sheets, avoiding complex libraries so that players can easily copy, authorize, and use the system without technical barriers.

Pitfalls to Consider
* Character Version Locking (A Feature, Not a Bug): The most important concept to communicate to your players. Characters are a snapshot of the rules at the time of their creation and will not receive updates or balance changes. This ensures stability and consistency at the cost of being on the latest ruleset.
* Storage Space: Since players will store a separate Core DB and Character Sheet template for each version they use, they will accumulate more files in their Google Drive over time. This is a minor issue but worth noting.
* Script Execution Time & Error Handling: Copying large amounts of data can be slow. Keep your database files reasonably sized. Scripts must use try...catch blocks to handle errors gracefully (e.g., if a user deletes a file) and write a clear message to the Log sheet instead of crashing.

Flex Project Architecture Summary
Our plan is to build a robust and maintainable system for the Flex TTRPG using a hybrid approach that combines a central code library with lightweight, local scripts.
Core Components & Philosophy
* Central Library (FlexLib): The vast majority of the game logic, all shared functions, and master error handling will reside in a single, standalone Google Apps Script project named FlexLib. This will be the "engine" of the entire system.
* Local Scripts as Triggers: The scripts attached to the player-facing Google Sheets (like the Player's Codex and Character Sheet) will be minimal. Their primary purpose is to be simple "triggers" that call the powerful functions within FlexLib. This keeps the local code clean and easy to manage.
* VS Code & clasp: The entire multi-part project (FlexLib, Player's Codex script, etc.) will be managed locally in a single VS Code project folder using clasp. This allows for professional version control with Git/GitHub and a superior development environment.
* Seamless Player Experience: This library-based architecture will not create any extra authorization steps or complexity for the end-user. From the player's perspective, the setup process will be identical to if all the code were stored locally.
Implementation Details
* Global Constants (g object): We will define a global constants object (e.g., g) inside the FlexLib project. This will serve as the single source of truth for game constants (like max level, sheet names, etc.) and will be accessible to all other scripts that include the library.
* Organized Codebase: The code within the FlexLib project itself will be organized into multiple .gs or .js files for clarity (e.g., Menus.gs, CharacterSheetLogic.gs, Globals.gs). clasp will automatically bundle these files together when pushing to Google's servers.

Flex Project Architecture Summary
This document outlines the architecture for the Flex TTRPG project, which is built on Google Sheets and powered by Google Apps Script.

   * Robust User Experience (UX): The system prioritizes clear, non-intrusive feedback. Functions provide immediate toast notifications (fShowToast) to confirm they are running, and conclude with clear success or error messages (fShowMessage).
________________


Key Architectural Patterns
   * Performance via Caching (fGetSheetData): All spreadsheet data reads are funneled through the fGetSheetData gatekeeper function. It uses a session cache by default for performance and supports a forceRefresh option for reading the absolute latest user input when necessary.
   * System Resilience (Self-Healing): The system is designed to recover from common user errors.
   1. Folder access uses fGetSubFolder to find critical folders even if the main project folder is moved or renamed.
   2. File access uses fGetVerifiedLocalFile to automatically restore master templates if a player accidentally deletes them.
   * Data Segregation: For complex user-generated content (like custom powers), we use a two-sheet system:
   1. A player-facing "working" sheet (e.g., <Powers>) where they input and edit data.

   * Development Environment: The entire project is managed locally in VS Code and synced with Google's servers using clasp. Version control is handled through Git/GitHub.
   * Global g Object: A global constants object (g) in FlexLib serves as the single source of truth for IDs, version numbers, and other constants.
